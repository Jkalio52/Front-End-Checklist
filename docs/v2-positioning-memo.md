# Frontend Checklist V2 Positioning Memo

## Summary

Frontend Checklist V2 should be positioned as a **trusted front-end quality system for humans and AI agents**, not as a generic library of tips.

That means the primary story is:

- a high-quality rule corpus
- executable workflows for launch, review, and audits
- MCP and skills as distribution for AI-assisted work
- guides and checklists as onboarding paths for humans

The project is strongest when it feels like shared infrastructure for front-end quality, not like a content site with many articles.

## Core Positioning

### Positioning statement

Frontend Checklist is the open-source front-end quality system for teams that want trusted review standards for both humans and AI agents.

### One-sentence homepage version

Trusted front-end quality rules for humans and AI agents.

### Expanded product version

Browse the rules on the web, run curated workflows, or connect the same corpus to MCP-capable tools for structured AI reviews.

## Who It Is For

### Primary audiences

- Senior engineers and tech leads who want consistent review standards
- AI-heavy developers who want better retrieval than generic LLM knowledge
- Agencies and consultancies that need repeatable audit workflows
- Developer educators who want examples, verification steps, and structured guidance

### Secondary audiences

- Junior developers learning what “good” looks like in production
- Product teams preparing launches or audits
- Open-source maintainers who want a shared quality checklist

### Audience fit by role

- Juniors need guided entry points, not the raw corpus first
- Seniors want search, prioritization, MCP, and review workflows
- AI users want structured tools, prompts, and proof that the corpus is opinionated and maintained

## Why It Matters In The AI Era

The differentiator is not “frontend best practices.” That is crowded.

The differentiator is:

- the rules are structured enough for retrieval and automation
- the corpus has evidence, scoring, and validation
- MCP lets the same rules drive real agent workflows
- skills make the content reusable in agentic tooling

This makes Frontend Checklist more than documentation. It becomes an execution layer for front-end quality.

## Message Architecture

### What to emphasize

- Trusted and maintained corpus
- Shared standards for teams and agents
- Actionable workflows, not just reference material
- Verification and remediation guidance
- Open source plus AI-native distribution

### What to de-emphasize

- “Huge list of best practices”
- generic “learn web development” framing
- counts as the primary value story
- broadness without guidance on where to start

### Product proof points

- hundreds of production-oriented rules
- hosted MCP server
- curated checklists for concrete workflows
- rule scoring, structure validation, and source validation
- guides tied back to rules and checklists

## Homepage Recommendations

### Recommended hero direction

Lead with trust, execution, and AI relevance in one sentence.

Suggested supporting copy:

- Browse the rule corpus on the web
- Start with curated workflows for launch and audits
- Connect the same corpus to AI tools through MCP

### Headline options

1. Trusted front-end quality rules for humans and AI agents
2. The open-source front-end quality system for modern teams
3. Front-end review standards your team and AI tools can actually use
4. A shared front-end quality layer for code reviews, audits, and launches
5. Production-ready front-end standards, executable by humans and AI

### Subhead options

1. Browse 385 rules across 11 categories, run curated workflows, or connect the corpus to MCP-capable tools for structured AI reviews.
2. Use the same rule corpus for launch checklists, code reviews, and AI-assisted audits.
3. Turn front-end best practices into repeatable workflows, trusted references, and AI-native review tools.

## Audience Entry Points

The homepage and docs should create clear entry routes instead of dropping everyone into the same giant corpus.

Recommended entry points:

- Start here as a junior developer
- Use this for senior code review
- Run a launch-readiness workflow
- Audit with AI using MCP
- Browse accessibility and performance first

These can be delivered through curated checklists, guide cards, and homepage sections before adding more complex product UI.

## What To Build Next

### Short-term

- Unify public counts and category metadata everywhere
- Make privacy and internationalization feel first-class in category UX
- Tighten homepage and MCP copy around the “quality system” story
- Add stronger “who this is for” entry points

### Medium-term

- Show real MCP demo flows with before/after review examples
- Add framework-specific workflows for React, Next.js, Astro, and static sites
- Package launch, accessibility, and performance as the canonical starting workflows

## Risks

- If counts drift across README, site, and metadata, the project feels less trustworthy
- If the homepage keeps framing the product as a giant checklist, the AI-native story gets buried
- If juniors do not get guided entry points, the corpus feels overwhelming
- If there are no public examples of the MCP server doing useful work, the AI story feels hypothetical

## Recommendation

Lean harder into the second identity: **Frontend Checklist as trusted front-end quality infrastructure**.

That is the version with the most relevance, the most differentiation, and the highest chance of earning attention beyond the usual “best practices” audience.
